# [PROJECT_NAME] 章程

## 核心原则

### I. Monorepo + pnpm 包管理（非协商）

每个功能都从 monorepo 结构开始；前端和后端分别有独立的工作区；项目必须使用 pnpm 作为包管理工具；禁止使用其他包管理工具。

**理由**：统一包管理避免依赖混乱，monorepo 结构便于代码共享和统一构建。

### II. React 编译优化（非协商）

所有 React 代码必须使用 React Compiler 进行优化；组件必须使用函数式组件 + React Hooks；禁止使用类组件。

**理由**：React Compiler 自动优化渲染性能，函数式组件 + Hooks 是现代 React 最佳实践。

### III. Drizzle ORM 强制执行（非协商）

所有数据库操作必须通过 Drizzle ORM；禁止使用裸 SQL；所有用户输入必须做服务端验证，使用 Zod。

**理由**：ORM 防止 SQL 注入等安全风险，Zod 提供强类型运行时验证。

### IV. API 规范与错误处理（非协商）

每个 API 端点必须满足 OpenAPI 规范；必须有错误处理和适当的 HTTP 状态码；提交信息使用 Conventional Commits 规范；代码风格遵循 Vite Plus 配置。

**理由**：OpenAPI 规范确保 API 可预测和可文档化，适当的错误处理提升开发者体验和系统可靠性。

### V. 前端安全、可访问性与测试（非协商）

前端页面安全参考 OWASP Top Ten；前端可访问性参考 WCAG AA 规范（对比度 4.5:1）；前端页面需要做组件测试；单元测试必须包含待测对象、场景和结果预期；每个单元测试必须独立，不存在共享外部因素，不访问外部上下文。

**理由**：安全漏洞、可访问性问题和缺乏测试是生产环境的主要风险来源。

## 项目约定

### 项目架构

- 项目是 PWA 应用，必须按照 PWA 规范生成相关文件和代码，并支持 PWA 应用各项必需的特性。
- 以 Clean Architecture 和 DDD 为核心，组织整体项目的层次结构；后端结合两者规范划分层次；前端作为展示层；数据库和 ORM 作为基础设施层。
- 前后端可复用的算法和逻辑、前端可复用的组件和 hooks 等代码，置于相应工作区的共享目录。

### 环境配置

- 环境变量使用 `.env.local`，提供 `.env.example` 模板。

### 容器化

- 支持 Docker，添加 Dockerfile 用于生成容器镜像。
- 使用数据库、缓存或文件存储时，按需拉取相应的镜像，运行相应的容器。
